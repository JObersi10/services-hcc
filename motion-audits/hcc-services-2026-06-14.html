Overall

The motion foundation is genuinely solid — spring curves throughout, GPU-safe animations, prefers-reduced-motion handled in both CSS and JS, a satisfying page-transition wipe, and a hero glow parallax that adds tactile depth. The gaps are expressiveness: an infinite opacity blink on the live dot is the single most recognisable AI-slop motion tell, and every card on the page enters identically regardless of hierarchy. Fixing those two — plus staggering the process icon path draws — lifts the motion quality from "competent" to "distinctly ours."

CriticalJhey · Jakub

Infinite blink pulse on the hero live dot

WhatThe .hero-eyebrow-dot loops opacity 1→0.3→1 at 1.4s forever via @keyframes blink. This is the single most recognisable AI-slop motion pattern — any infinite opacity pulse on a status indicator reads as generated, not designed.

Why it mattersJhey: "what could this become?" A live dot should pop in with spring personality and settle. The infinite loop is lazy; the spring settle is expressive. Jakub: "if users consciously notice the animation it's too much" — and they will, every page load, forever.

Recommended motionReplace @keyframes blink infinite with a one-shot spring-pop using animation-fill-mode: forwards. The dot appears, overshoots to 118% scale, snaps back, and settles. Demo shows the spring pop (looping here for preview only — in production use forwards).

CriticalJakub · Jhey

Uniform 0.9s reveal on 20+ elements — zero visual hierarchy

WhatEvery .reveal element uses identical opacity:0 + translateY(28px) + blur(4px) at 0.9s — hero stats, service cards, testimonials, work cards, section labels, all the same. Blur and spring curve are correct; the uniformity erases hierarchy.

Why it mattersJakub: "using the same animation everywhere" kills context-sensitivity. Utility labels should be faster (300ms, 8px) than headings (450ms, 16px) than card grids (600ms, 24px). Right now nothing feels intentional — it's all one flat reveal plane.

Recommended motionKeep the blur+spring recipe, vary duration and translateY by element tier. The JS stagger system (80ms per sibling) already works — just tighten the base durations. Add a CSS custom property --reveal-dur per element type so the three tiers can be set in one place.

ImportantJakub

All card hovers use identical translateY(-3px)

WhatService cards, testimonials, work cards, team cards all hover with transform: translateY(-3px). Jakub's rule: "context should drive timing and movement choices" — a featured bento card and a supporting testimonial should not feel identical on hover.

Why it mattersThe user can't feel the difference in importance between a primary card and a supporting one. The bento layout creates visual hierarchy via sizing — the hover should reinforce it, not flatten it.

Recommended motionFeatured/primary cards: translateY(-7px) scale(1.01) + stronger shadow. Supporting cards: translateY(-2px), no scale. Work thumbnail cards (image-heavy): no lift, just border glow. Demo shows featured vs supporting contrast.

ImportantJhey · Jakub

Process icon paths draw simultaneously — stagger wasted

WhatAll paths in each process SVG icon receive the same stroke-dashoffset transition simultaneously. The icon has 2–4 paths; they all appear at once instead of assembling sequentially.

Why it mattersThe sequential line-draw is the most expressive moment in the process section. Drawing all paths at the same frame collapses a rich "building" feeling into a simple reveal. This is the single highest-impact motion upgrade available for minimal code change.

Recommended motionAdd transitionDelay per path index in the JS animation loop. Each subsequent path gets +180ms delay, so the icon assembles itself piece by piece. 4 paths = 0ms, 180ms, 360ms, 540ms. Demo shows 4-path sequential draw.
